# 扩展已有 Linear 项目

本 Skill 用于将新需求纳入 Linear 已有项目。目标不是简单"追加几个 issue"，而是识别新增需求对项目目标、范围、架构、里程碑、依赖和风险的影响，并保留项目演进的可追溯性。

---

## 1. 触发条件

- "根据 Linear 已有 project 完善新的需求"
- "给这个 project 增加需求"
- "调整项目说明"
- "新增里程碑 / issues / 依赖关系"
- "这个项目现在要加入一个新模块"
- "把这段需求并入现有 Linear 项目"

---

## 2. 最终产出

1. **当前项目理解**：项目目标、当前范围、现有 Milestones、现有 Issues、最近状态更新、已有风险与依赖。
2. **新需求澄清结果**：用户、场景、目标、与现有范围的关系、是否属于 MVP/vNext/Later、新增约束/非目标/成功指标。
3. **影响分析**：对项目说明、里程碑、Issue、架构、时间、依赖、风险的影响。
4. **Linear 更新**：更新 Project description、新建/更新 Milestones、规划/调整 Cycles、新建 Issues、建立依赖关系、发布/起草状态更新。

---

## 3. 读取已有 Project

在写入前必须读取或尝试读取：

- Project 基本信息（名称、状态、描述、目标日期、负责人、成员）
- Milestones（名称、描述、目标日期、进度）
- Issues（标题、描述、状态、优先级、负责人、里程碑、cycle、labels）
- Cycles（当前/下一 Cycle、时间窗口、已纳入 Issues、完成率，如工具支持）
- Issue relations（blocked / blocking / related / duplicate）
- Project dependencies
- 最近 project updates

如果无法读取 Linear：

> 我暂时无法读取该 Linear Project。请提供 Project 当前说明、里程碑、核心 Issues 或截图/导出内容；否则我只能基于你提供的信息生成"待写入方案"，不能声称已完成更新。

---

## 4. 新需求多轮澄清

### 4.0 核心原则

**扩展项目与新建项目一样，必须先充分理解需求再动手。**
不要在用户只说了一句"加个需求"时就直接创建 Issue。
每轮最多问 5 个问题。优先问会改变范围、架构、里程碑或依赖的问题。

### 4.1 第 1 轮：需求定位（必问）

展示当前项目基线摘要后，再问：

```
我已读取了项目 [项目名] 的当前状态。在把新需求纳入前，我需要确认：

1. 这个新需求服务哪个用户或场景？与现有项目目标的关系是什么？
2. 它是必须在当前周期完成（MVP），还是可以放到后续（vNext/Later）？
3. 你期望的交付结果是什么？怎样算"做完了"？
4. 如果不做这个需求，项目是否仍可按计划推进？
5. 你目前对这个需求的想法有多成熟？（有具体方案 / 只有大方向 / 只有问题没有方案）
```

### 4.2 第 2 轮：范围与边界（按需）

如果第 1 轮信息不足以判断范围和影响：

```
为了准确评估影响和拆分 Issue，我还需要确认：

1. 这个需求包含哪些必须做的能力？有没有明确不做的边界？
2. 它依赖哪些现有模块、数据、API 或第三方系统？
3. 是否需要新增数据实体、API 端点或权限？
4. 是否有外部约束？例如与其他项目的联动、截止日期、当前/下一 Cycle 容量、工具链限制。
5. 这个需求是否改变现有的项目成功指标或上线标准？
```

### 4.3 第 3 轮：技术影响确认（按需）

如果新需求可能影响系统设计：

```
这个需求可能影响架构，我需要进一步确认：

1. 是否新增或修改数据实体、数据库 schema、存储策略？
2. 是否新增 API、Webhook、事件、命令行入口或定时任务？
3. 是否影响已有用户流程、状态机或核心算法逻辑？
4. 是否需要新的部署组件、容器、队列或环境变量？
5. 是否涉及安全、隐私、数据迁移或向后兼容？
```

### 4.4 澄清充分的判定

当以下内容可以被明确写出时，即可进入影响分析阶段：

- 一句话需求目标。
- 与现有项目目标的关系（增强 / 变更 / 独立 / 不相关）。
- 涉及的用户和场景。
- 必须做和不做的边界。
- 对架构、里程碑、依赖和风险的初步判断。

如果用户要求"先按你的判断来"，可以继续，但必须把缺失信息写为假设和待确认项。

### 4.5 确认方案再执行

完成澄清和影响分析后，必须向用户展示完整的 Linear 更新方案，包括：

- 新建/更新的 Issue 列表（含标题、描述摘要、labels、milestone、依赖）
- 对项目说明的更新摘要
- 新增/调整的 Milestone
- 当前/下一 Cycle 的纳入与顺延计划（如适用）
- 新增的依赖关系
- 项目状态更新草案

**用户确认后再执行写入。** 不得在用户未确认时直接创建 Issue。

---

## 5. 新需求分类

将新需求归类为以下之一：

### 5.1 范围内增强

- 与当前目标一致，不改变里程碑主路径，架构影响较小
- 处理：更新项目说明范围，新建 1-3 个 Issue，状态保持 `onTrack`

### 5.2 MVP 范围变更

- 改变 MVP 必须交付内容，影响上线时间或核心验收
- 处理：更新 MVP 范围/非目标/成功指标，新增/调整 Milestones 和 Cycle 计划，标记风险，状态改为 `atRisk`

### 5.3 架构级变更

- 新增核心模块、数据模型、权限模型、API 边界或部署组件
- 处理：输出架构影响分析，新增/更新 ADR，更新模块划分，建立更严格依赖关系

### 5.4 应拆分为新项目

- 目标用户/成功指标/上线周期明显不同，独立交付价值高
- 处理：建议创建新 Project，在现有项目中创建 related/dependency

### 5.5 应拒绝或延期

- 与项目目标不一致，ROI 低，证据不足，范围蔓延
- 处理：不创建开发 Issue，在项目说明中加入"已评估但暂不纳入"

---

## 6. 影响分析

在写入前输出影响分析。详细模板见 [references/templates.md](references/templates.md)。

核心结构：

1. 当前项目基线（目标、MVP、里程碑、关键风险、阻塞）
2. 新需求摘要（需求、用户、场景、预期结果、紧急程度）
3. 归类结果
4. 对范围的影响（新增/调整/不纳入）
5. 对架构的影响（模块/数据模型/API/部署/安全/可观测性）
6. 对 Linear 计划的影响（Project description/Milestone/Cycle/Issue/Dependencies/Update）
7. 风险与决策

---

## 7. Project description 更新规则

更新项目说明时不要覆盖历史信息。使用"变更记录"和"当前版本范围"：

```markdown
## 变更记录
| 日期 | 变更 | 原因 | 影响 | 决策 |
|---|---|---|---|---|
| [YYYY-MM-DD] | [新增/调整] | [原因] | [影响] | [决策] |

## 当前 MVP 范围
[更新后的范围]

## 新增需求说明
### [需求名称]
- 用户：
- 场景：
- 目标：
- 范围：
- 非目标：
- 成功指标：
- 架构影响：
- 依赖：
- 风险：
```

如果项目说明原本没有结构，先保留原文，再追加结构化版本。

---

## 8. ADR 更新规则

新增需求引发以下变化时，必须新增或更新 ADR：

- 架构风格变化
- 新增核心存储或索引策略
- 新增外部系统集成
- 新增异步队列、事件机制、定时任务
- 权限模型变化
- 数据合规/审计策略变化
- 部署拓扑变化
- API 契约变化且影响调用方

---

## 9. 新建/调整 Milestones

**何时新建**：新需求有独立阶段性交付物、跨越多个 Issue、有独立上线窗口、引入高风险集成。

标题格式：`M[编号]｜[新增需求名]：[阶段目标]`

**何时调整已有**：新需求只是补充现有阶段目标、不改变主路径、不需要独立验收窗口。

---

## 10. Cycle 调整

当新增需求影响当前或下一执行窗口时，必须输出 Cycle 影响：

- **纳入当前 Cycle**：只在需求已澄清、依赖可解除、验收标准明确且容量允许时执行。
- **放入下一 Cycle**：适用于当前 Milestone 相关但本周期容量不足的 Issue。
- **不进入 Cycle**：适用于 vNext/Later、依赖未解除或范围不清的 Issue。
- **移出当前 Cycle**：属于范围变更或容量收缩，必须说明顺延原因，并在用户确认后写入。

Cycle 不替代 Milestone；Issue 仍必须归属 Milestone。

---

## 11. 新建 Issues

### 标题格式

- 新增：`[需求名]：[可交付结果]`
- 调整：`[原模块]：支持[新增能力]`
- 风险/决策：`【风险】[名]：[阻塞]` / `【决策】[名]：[取舍]`

Issue 描述模板见 [references/templates.md](references/templates.md)。

新建或调整 Issue 时，如果该 Issue 被纳入当前/下一执行窗口，应同时规划 Cycle 归属；如果不纳入，应说明顺延原因。

### Label 规则

本 workspace 使用两个 Label Group：

#### Task-difficulty（必选，每个 Issue 必须有且仅有一个）

| Label | 适用场景 |
|---|---|
| `High-difficulty` | 需要深度逻辑推理、系统性问题解决、大量上下文处理的复杂任务 |
| `Medium-difficulty` | 需要多步逻辑和中等上下文理解的标准任务 |
| `Low-difficulty` | 上下文隔离、逻辑简单的直接任务 |

判断维度：涉及模块数量、跨系统依赖、算法复杂度、状态管理复杂度、调试难度。

#### Area（可选，根据 Issue 涉及的工程领域选择）

| Label | 适用场景 |
|---|---|
| `Backend` | 服务端计算、数据库、API 开发、核心系统架构 |
| `Frontend` | 用户界面、客户端逻辑、视觉样式、用户交互 |

一个 Issue 可以同时打 Backend 和 Frontend（如全栈任务）；也可以不打 Area（如纯需求/文档/项目管理类 Issue）。

#### 写入规则

- 创建 Issue 时必须在 `labels` 字段包含一个 Task-difficulty label。
- 根据 Issue 内容判断是否需要 Area label，如需则一并写入。
- 已有 Issue 的 labels 字段是 append-only，更新时不要覆盖已有 label。
- 已有独立 label（Bug、Feature、Improvement 等）继续保留，不受 Label Group 规则影响。

---

## 12. 依赖关系更新

默认建立规则：

- 新数据模型 Issue `blocks` 新 API/新核心模块
- 新权限或合规 Issue `blocks` 上线相关 Issue
- 第三方集成 Issue `blocked by` 第三方账号、接口权限、测试环境
- 迁移 Issue `blocks` 上线
- 架构决策/ADR Issue `blocks` 受影响开发 Issue

每条依赖必须说明：依赖原因、解除条件、风险等级。

---

## 13. 写入流程

```text
1. 读取 Project、Milestones、Cycles、Issues、relations、updates
2. 总结当前项目基线
3. 澄清新需求
4. 输出影响分析
5. 生成更新计划
6. 如果用户已授权写入：
   6.1 更新 Project description
   6.2 新建/更新 Milestones
   6.3 定位或调整 Cycle（仅在工具支持且用户授权时）
   6.4 新建/更新 Issues，并按需设置 cycle
   6.5 建立 issue relations / project dependencies
   6.6 校验每个 Issue 的 Task-difficulty label 覆盖率，缺失则补充
   6.7 发布 project update
   6.8 回读校验
7. 输出写入结果和待确认项
```
